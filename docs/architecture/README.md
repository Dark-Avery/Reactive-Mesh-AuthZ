# Архитектура

Reactive Mesh AuthZ как исследовательский стенд состоит из следующих компонентов:

- `receiver/cpp` — C++ HTTP-приёмник событий класса SSF/CAEP/SET;
- `envoy-custom/reactive_pep` — C++ HTTP-фильтр Envoy с учётом активных потоков и реактивным завершением;
- `baseline/opa/policy.rego` и `baseline/opa/authz_service.cpp` — контур сравнения `OPA + Envoy` и адаптеры для внешних архитектур сравнения;
- `grpc/cpp` — демонстрационные C++ gRPC-сервер и клиент;
- `Redis` — быстрый слой состояния отзыва и запрета с TTL.

## Модель идентификаторов

- новые gRPC-потоки связываются по `sub`, `sid` и `jti` через заголовки `x-sub`, `x-sid` и `x-jti`;
- приёмник записывает состояние запрета для каждого присутствующего идентификатора;
- reactive PEP завершает любой активный поток, совпавший хотя бы по одному из этих идентификаторов;
- архитектуры сравнения запрещают создание нового потока при повторном запросе после отзыва.

## Модель приёма событий

- базовый стендовый путь использует plain JSON через `POST /event`;
- подписанный путь использует compact JWS `HS256`;
- локальный OSS IdP выдаёт OIDC/JWKS bearer token, из которого извлекаются `sub`, `sid` и `jti`.

## Автомат состояний

Основной автомат имеет вид:

`New -> Active -> RevokedPending -> Terminated`

или, если допуск запрещён ещё до создания потока:

`New -> Denied`
